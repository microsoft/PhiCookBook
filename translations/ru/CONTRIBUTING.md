# Участие в проекте

Этот проект приветствует участие и предложения. Большинство вкладов требуют от вас согласия с Соглашением о лицензировании вкладчика (CLA), подтверждающим, что вы имеете право и действительно предоставляете нам права на использование вашего вклада. Для подробностей посетите [https://cla.opensource.microsoft.com](https://cla.opensource.microsoft.com).

Когда вы отправляете запрос на слияние (pull request), бот CLA автоматически определит, нужно ли вам предоставить CLA, и соответствующим образом отметит PR (например, проверкой статуса, комментарием). Просто следуйте инструкциям, предоставленным ботом. Вам нужно будет сделать это только один раз для всех репозиториев, использующих наше CLA.

## Кодекс поведения

Этот проект принял [Кодекс поведения Microsoft Open Source](https://opensource.microsoft.com/codeofconduct/).  
Для получения дополнительной информации ознакомьтесь с [Часто задаваемыми вопросами о Кодексе поведения](https://opensource.microsoft.com/codeofconduct/faq/) или свяжитесь с [opencode@microsoft.com](mailto:opencode@microsoft.com) для любых дополнительных вопросов или комментариев.

## Предостережения по созданию задач

Пожалуйста, не создавайте задачи в GitHub для общих вопросов поддержки, так как список задач GitHub предназначен для запросов на добавление функций и отчетов об ошибках. Это позволит нам легче отслеживать реальные проблемы или ошибки в коде и отделять общие обсуждения от работы над кодом.

## Как внести вклад

### Руководство по запросам на слияние (Pull Requests)

При отправке запроса на слияние (PR) в репозиторий Phi-3 CookBook, пожалуйста, следуйте следующим рекомендациям:

- **Форк репозитория**: Всегда создавайте форк репозитория в своем аккаунте перед внесением изменений.

- **Разделяйте запросы на слияние (PR)**:
  - Отправляйте каждый тип изменений в отдельном PR. Например, исправления ошибок и обновления документации должны быть отправлены в разных PR.
  - Исправления опечаток и незначительные обновления документации можно объединять в один PR, если это уместно.

- **Разрешение конфликтов слияния**: Если ваш запрос на слияние показывает конфликты, обновите свою локальную ветку `main`, чтобы она соответствовала основному репозиторию, перед внесением изменений.

- **Отправка переводов**: При отправке PR с переводом убедитесь, что папка перевода содержит переводы всех файлов из оригинальной папки.

### Руководство по переводам

> [!IMPORTANT]
>
> При переводе текста в этом репозитории не используйте машинный перевод. Беритесь за переводы только для тех языков, в которых вы уверенно владеете.

Если вы уверенно владеете языком, отличным от английского, вы можете помочь с переводом контента. Чтобы ваш вклад был правильно интегрирован, следуйте этим рекомендациям:

- **Создайте папку для перевода**: Перейдите в соответствующую папку раздела и создайте папку перевода для языка, на который вы переводите. Например:
  - Для раздела "введение": `PhiCookBook/md/01.Introduce/translations/<language_code>/`
  - Для раздела "быстрый старт": `PhiCookBook/md/02.QuickStart/translations/<language_code>/`
  - Продолжайте этот шаблон для других разделов (03.Inference, 04.Finetuning и т.д.).

- **Обновите относительные пути**: При переводе добавьте `../../` в начало относительных путей внутри markdown-файлов, чтобы ссылки работали правильно. Например, измените следующим образом:
  - Измените `(../../imgs/01/phi3aisafety.png)` на `(../../../../imgs/01/phi3aisafety.png)`.

- **Организуйте ваши переводы**: Каждый переведенный файл должен быть размещен в соответствующей папке перевода раздела. Например, если вы переводите раздел "введение" на испанский, создайте следующую структуру:
  - `PhiCookBook/md/01.Introduce/translations/es/`.

- **Отправляйте полный PR**: Убедитесь, что все переведенные файлы для раздела включены в один PR. Мы не принимаем частичные переводы для раздела. При отправке PR с переводом убедитесь, что папка перевода содержит переводы всех файлов из оригинальной папки.

### Руководство по написанию

Чтобы обеспечить единообразие во всех документах, следуйте этим рекомендациям:

- **Форматирование URL**: Оборачивайте все URL в квадратные скобки, за которыми следуют круглые скобки, без лишних пробелов внутри или вокруг. Например: `[example](https://www.microsoft.com)`.

- **Относительные ссылки**: Используйте `./` для относительных ссылок, указывающих на файлы или папки в текущем каталоге, и `../` для ссылок в родительском каталоге. Например: `[example](../../path/to/file)` или `[example](../../../path/to/file)`.

- **Локали, не привязанные к странам**: Убедитесь, что ваши ссылки не содержат локали, привязанные к конкретным странам. Например, избегайте `/en-us/` или `/en/`.

- **Хранение изображений**: Храните все изображения в папке `./imgs`.

- **Описательные имена изображений**: Присваивайте изображениям описательные имена, используя английские буквы, цифры и дефисы. Например: `example-image.jpg`.

## Рабочие процессы GitHub

Когда вы отправляете запрос на слияние, запускаются следующие рабочие процессы для проверки изменений. Следуйте приведенным ниже инструкциям, чтобы ваш запрос на слияние прошел проверки рабочих процессов:

- [Проверка сломанных относительных путей](../..)
- [Проверка отсутствия локали в URL](../..)

### Проверка сломанных относительных путей

Этот рабочий процесс проверяет, что все относительные пути в ваших файлах правильные.

1. Чтобы убедиться, что ваши ссылки работают правильно, выполните следующие действия с помощью VS Code:
    - Наведите курсор на любую ссылку в ваших файлах.
    - Нажмите **Ctrl + Click**, чтобы перейти по ссылке.
    - Если вы нажмете на ссылку, и она не работает локально, это вызовет рабочий процесс и не будет работать на GitHub.

1. Чтобы исправить эту проблему, выполните следующие действия с использованием предложений путей, предоставляемых VS Code:
    - Введите `./` или `../`.
    - VS Code предложит вам выбрать из доступных вариантов на основе введенного текста.
    - Следуйте по пути, нажав на нужный файл или папку, чтобы убедиться, что ваш путь правильный.

После добавления правильного относительного пути сохраните и отправьте изменения.

### Проверка отсутствия локали в URL

Этот рабочий процесс проверяет, что ни один веб-URL не содержит локали, привязанной к стране. Поскольку этот репозиторий доступен глобально, важно, чтобы URL не содержали локали вашей страны.

1. Чтобы убедиться, что ваши URL не содержат локали, выполните следующие действия:

    - Проверьте наличие текста вроде `/en-us/`, `/en/` или любой другой языковой локали в URL.
    - Если этого нет в ваших URL, вы пройдете эту проверку.

1. Чтобы исправить эту проблему, выполните следующие действия:
    - Откройте путь к файлу, указанный рабочим процессом.
    - Удалите локаль страны из URL.

После удаления локали сохраните и отправьте изменения.

### Проверка сломанных URL

Этот рабочий процесс проверяет, что все веб-URL в ваших файлах работают и возвращают код состояния 200.

1. Чтобы убедиться, что ваши URL работают правильно, выполните следующие действия:
    - Проверьте статус URL в ваших файлах.

2. Чтобы исправить сломанные URL, выполните следующие действия:
    - Откройте файл, содержащий сломанный URL.
    - Обновите URL на правильный.

После исправления URL сохраните и отправьте изменения.

> [!NOTE]
>
> В некоторых случаях проверка URL может завершиться сбоем, даже если ссылка доступна. Это может произойти по нескольким причинам, включая:
>
> - **Сетевые ограничения**: Серверы GitHub Actions могут иметь сетевые ограничения, препятствующие доступу к определенным URL.
> - **Проблемы с тайм-аутом**: URL, которые отвечают слишком долго, могут вызвать ошибку тайм-аута в рабочем процессе.
> - **Временные проблемы сервера**: Временные отключения или техническое обслуживание серверов могут сделать URL временно недоступным во время проверки.
